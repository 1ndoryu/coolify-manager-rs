## 145A-1 — fix_db_auth: soporte legacy + diagnóstico DNS collision

- **Que:** Refactor completo de `fix-db-auth` para detectar automáticamente el esquema de credenciales
  (nuevo: `SERVICE_PASSWORD_POSTGRES`/`rust_app`/`rust_db` vs legacy: `DB_PASSWORD` + parseo de DATABASE_URL).
  Motivado por incidente restaurante.wandori.us donde `28P01` era causado por DNS collision, no mismatch de contraseña.
- **Contexto del incidente:**
  - App stack `b8s0cks444o0sogo8kg8wcgw` conectada a red `coolify` (Traefik) + red del stack simultáneamente
  - `coolify-db` tiene alias `postgres` en la red `coolify` → Docker resuelve 10.0.1.2 en vez de 10.0.10.4
  - Fix manual en VPS: `@postgres:` → `@postgres-b8s0cks444o0sogo8kg8wcgw:` en docker-compose.yml
  - Templates nuevos ya usan `postgres-{{STACK_UUID}}` — stacks nuevos seguros
- **Archivos modificados:**
  - `src/commands/fix_db_auth.rs` — añadido `detect_db_credentials()`, `extract_user_db_from_compose()`,
    actualizado `test_postgres_auth()` y ALTER USER para usar variables detectadas
- **Gotchas:**
  - `28P01` es diagnóstico engañoso: puede ser contraseña incorrecta O container equivocado
  - Verificar siempre con `nslookup postgres` desde el container antes de tocar contraseñas
  - `fix_database_url_hostname()` ya existía y corrige `@postgres:` → `@postgres-{uuid}:` automáticamente;
    ahora se llama incluso cuando auth ya está OK para corregir compose proactivamente
- **Sentinel:** No aplica
- **GLORY:** No aplica — lógica específica de coolify-manager-rs
